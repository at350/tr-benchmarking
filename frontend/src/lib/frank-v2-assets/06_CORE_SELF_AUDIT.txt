CORE SELF-AUDIT CHECKLIST
For Gold-Packet Drafting Across Doctrine Packs

PURPOSE
This file is a cross-pack drafting self-audit tool for benchmark answers, source-to-benchmark conversions, and reverse-engineered outputs. It is intentionally thin. It does not supply doctrine. Its job is to catch reusable structural, routing, source-fidelity, and answer-discipline failures before a draft is finalized.

SCOPE
Use this checklist across the core workflow and across doctrine packs.
Typical uses:
- common-law oral-promise / writing-enforceability pack
- land-contract pack
- executor/administrator personal-promise pack
- UCC sale-of-goods writing pack
- later packs that share the same benchmark workflow

If the selected doctrine pack or uploaded source contains pack-specific requirements, that pack or source controls the substance. This checklist should not be used to force one pack's doctrine into another.

USE RULES
- Use this after drafting but before release.
- Treat uploaded authority as the substantive source of law.
- Treat the core workflow/template and output-shape files as workflow authority.
- Treat the selected doctrine pack as doctrinal authority.
- Use this checklist to revise the draft, not to override the source or selected pack.
- If the source conflicts with this checklist, the source controls.
- If a major red flag remains after review, revise before finalizing.
- If the answer is being used for clustering or benchmarking, preserve the exact headings required by the current workflow/template.
- If the draft appears routed to the wrong doctrine pack, stop and reroute before polishing language.

OUTPUT-SHAPE DISCIPLINE
Preserve the exact headings required by the current workflow/template.
At minimum, the draft should clearly surface:
- jurisdiction assumption
- bottom-line outcome
- controlling doctrine
- transaction / formation characterization
- controlling trigger or writing requirement
- compliance / substitute / exception analysis
- other defenses or competing doctrines
- strongest counterargument
- bounded uncertainty

FAST TRIAGE
Run these checks first:

1. Bottom line + controller
- Does the draft clearly state who wins or whether the agreement/promise is enforceable?
- Does it explicitly name the likely controlling doctrine?
- Is the conclusion supported by the analysis?
- Is any uncertainty specific rather than generic?

2. Controlling trigger + ordered path
- Does the draft identify the controlling gate, rule, or trigger first?
- Does it use the correct test rather than a loose paraphrase?
- Are secondary issues kept secondary?
- If a writing doctrine is relevant, does the draft distinguish enforceability from proof?

3. Exception mapping
- Do fallback doctrines or substitutes appear only after the main gates?
- Does the draft explain what each workaround can and cannot cure?
- Does it avoid one-exception-cures-all reasoning?

4. Routing / pack fit
- Does the selected doctrine pack actually match the controlling issue?
- Is the draft borrowing doctrine, examples, or trigger tests from a different pack?
- If fit is mixed or weak, does the draft stop at source extraction + JD review flags instead of forcing a final benchmark answer?

If any fast-triage item fails, revise before moving on.

FULL SELF-AUDIT

MODULE 0 — METADATA SNAPSHOT
Use these to orient the draft before deeper review.

- Outcome stated: Enforceable / Not enforceable / Mixed / No clear conclusion
- Outcome confidence: Strong / Moderate / Weak
- Jurisdiction assumption stated: Yes / No
- Controlling doctrine named: Yes / No
- Selected pack fit: Clear / Mixed / Weak / Needs rerouting
- Benchmark posture: Narrow source-grounded / Generalizable only with supporting authority / Portable benchmark / Unclear

MODULE 1 — STRUCTURAL GATEKEEPING
A. Issue spotting + prioritization
Ask:
- Does the draft follow the ordered path required by the workflow: transaction or formation characterization -> controlling doctrine or primary trigger -> secondary issues if relevant -> fallback doctrines -> other defenses or competing doctrines -> conclusion?
- Does it identify the dispositive issue early?
- Does it avoid leading with fairness, estoppel, policy, or a doctrine laundry list?
- If multiple doctrines appear, does the draft explain which one controls and why?

Revise if you see:
- missed dispositive issue
- no prioritization
- fallback-first analysis
- laundry-list analysis with no path
- wrong-pack logic appearing as the main path

K. Barrier stacking + exception mapping
Ask:
- Does the draft keep independent barriers and doctrinal steps separate?
- Does it explain what each substitute, exception, or workaround can actually reach?
- Does it avoid collapsing controller, secondary issues, and fallback doctrines into one merged discussion?

Revise if you see:
- exception bleed-over
- one-exception-cures-all logic
- merged gates or collapsed doctrinal steps
- fallback theory swallowing the main gate

J. Clear bottom line + structured reasoning
Ask:
- Is there a clear bottom-line outcome?
- Is the conclusion tied to the controlling doctrine?
- Is there a real strongest counterargument?
- Is uncertainty bounded and specific rather than generic "it depends" language?

Revise if you see:
- no clear conclusion
- conclusion unsupported by the analysis
- no controlling doctrine named
- depends-boilerplate instead of specific uncertainty
- counterargument missing or straw-man

MODULE 2 — PRIMARY DOCTRINAL PATH DISCIPLINE
B. Transaction / formation characterization
Ask:
- Does the draft characterize the transaction, promise structure, or legal relationship only as much as the source and selected pack support?
- Does it distinguish preliminary characterization issues from fallback theories?
- Does it avoid assuming that performance, subject matter, or a label automatically resolves the characterization question?
- If a threshold characterization matters (for example, bargain vs condition, goods vs services, land-interest type, personal vs representative capacity, promise to creditor vs promise to debtor), is it handled correctly?

Revise if you see:
- threshold characterization omitted
- roles or transaction type mischaracterized
- any-act-is-enough reasoning
- formation and reliance collapsed
- unsupported assumption of primary liability, novation, or equivalent status shift

C. Controlling doctrine / trigger test
Ask:
- Does the draft identify the likely controlling doctrine, gate, or trigger first?
- Does it state the correct legal test for that controller?
- Does it distinguish unenforceability from proof difficulty when that distinction matters?
- Does it keep distinct categories or triggers separate rather than blending them together?

Revise if you see:
- wrong controller
- trigger test misstated
- proof-only treatment of an enforceability rule
- distinct categories or gates collapsed
- unsupported compliance or substitute theory inserted as though it were the main rule

D. Secondary issue discipline
Use only if genuinely relevant.
Ask:
- Are secondary issues addressed only if the facts or source actually trigger them?
- Are timing, quantity, addressee, capacity, description, sequence, or other threshold facts handled accurately?
- Are non-triggered secondary issues kept brief?

Revise if you see:
- secondary issue expanded when not actually triggered
- threshold direction or timing misstated
- quantity / description / addressee / capacity / sequence fact ignored
- side issue becomes main issue without source support

E. Compliance / satisfaction / substitute path
Use only if genuinely relevant.
Ask:
- If direct compliance matters, does the draft analyze the actual requirement rather than a vague substitute?
- If a substitute path matters, does the draft separate it from direct compliance?
- Does it avoid inventing writings, signatures, confirmations, admissions, creditor assent, possession, payment, manufacture facts, or other satisfaction facts not in the source?

Revise if you see:
- direct compliance requirement misstated
- substitute path treated as automatic
- invented satisfaction facts
- substitute doctrine used before the main requirement is identified

MODULE 3 — FALLBACK DOCTRINES, COMPETING DOCTRINES, AND DEFENSES
F. Fallback doctrines + limits
Ask:
- Are fallback doctrines discussed only after the main gates?
- Does the draft explain the limits of each workaround or alternative doctrine?
- Does it note any jurisdiction-specific limit when relevant?
- Does it avoid inserting irrelevant doctrines just because they sound equitable or complete?

Revise if you see:
- fallback treated as universal override
- no limits stated
- requirements missing
- irrelevant doctrine inserted as a major driver
- broad fairness language replacing doctrinal fit

G. Conduct-dependent substitute rigor
Use only if a fallback path depends on conduct, causation, reliance, prejudice, admissions, possession, payment, manufacture, or another fact-sensitive substitute theory.
Ask:
- Does the draft identify the exact conduct that matters?
- Does it analyze why that conduct qualifies under the selected theory?
- Does it avoid treating performance or later conduct as automatically dispositive?
- If causation or detriment matters, is that analyzed rather than assumed?

Revise if you see:
- substitute invoked with no element-by-element fit
- critical conduct not identified
- automatic cure reasoning
- causation, prejudice, or detriment assumed rather than analyzed

H. Other defenses / competing doctrines / conditions
Ask:
- Does the draft distinguish motive from an actual condition, excuse, or defense when relevant?
- Does it keep competing doctrines secondary unless the source or selected pack makes them central?
- Does it avoid inventing waiver, novation, misrepresentation, condition precedent, or other side doctrines not supported by the source?

Revise if you see:
- motive converted into a condition without support
- invented condition or defense
- excuse analysis displacing a clearer controlling doctrine
- competing doctrine imported from the wrong pack

MODULE 4 — CROSS-CUTTING ANSWER DISCIPLINE
I. Factual fidelity + internal consistency
Ask:
- Are all material facts drawn from the source or clearly framed as hypothetical additions?
- Are timing, quantity, sequence, and party roles accurate?
- Are there internal contradictions?

Revise if you see:
- invented facts
- timing / quantity / sequence error
- party-role confusion
- internal contradiction

L. Scope calibration / claim discipline
Ask:
- Does the draft avoid false universals?
- If jurisdictional or doctrinal uncertainty exists, is it identified specifically?
- Does the draft stay within the chosen benchmark posture?
- Are rule statements scoped to the source, stated assumption, or selected pack rather than overgeneralized?

Revise if you see:
- false universal rule statements
- vague jurisdiction hedging
- answer written more broadly than the source or benchmark posture supports
- cross-pack generalizations presented as settled law

M. Relevance discipline / prompt adherence
Ask:
- Does the draft stay focused on the task actually asked?
- Does it use the required headings?
- Does it use full doctrine names rather than unexplained abbreviations?
- Does it avoid practical advice, policy drift, remedies sprawl, or off-task discussion?

Revise if you see:
- missing required headings
- off-prompt content
- practical advice or litigation strategy dominating the draft
- remedies or policy discussion crowding out the controlling issue

QUESTION OUTPUT ADD-ON
Use this only when the draft being audited is a reverse-engineered question or variation.

Ask:
- Is the call of the question neutral?
- Does the question avoid naming or pre-answering the controlling doctrine?
- Are only the holding-driving facts and one or two realistic distractors preserved?
- Does the question avoid invented writings, signatures, party-role changes, jurisdiction drift, or timing changes that alter the analysis?

Revise if you see:
- answer leakage
- invented facts or documents
- changed legal roles
- too many equally weighted doctrinal paths
- question no longer matches the benchmark posture or selected pack

RED-FLAG OVERLAYS
These are not extra sections. They are stop-checks before release.

Stop and revise if any of these are present:
- controlling doctrine omitted
- wrong doctrine pack appears to be driving the answer
- material rule or trigger test misstatement
- material fact, timing, quantity, role, or sequence error
- exception bleed-over
- invented writing, signature, assent, confirmation, admission, payment, or other compliance fact
- irrelevant doctrine used as a major driver
- excessive hedging in place of analysis
- source too weak or split-dependent for a stable benchmark answer, but the draft is written as settled
- jurisdiction-specific rule imported without stating the jurisdictional assumption

FINAL RELEASE CHECK
Do not finalize unless all of these are true:
- A clear bottom-line outcome appears early.
- The likely controlling doctrine is named.
- The selected pack still fits the issue.
- The draft follows the ordered path.
- The controlling trigger is analyzed before fallback doctrines.
- Independent barriers remain separate.
- Substitutes or exceptions are mapped only to the barriers they can actually address.
- Non-triggered secondary issues are brief.
- No invented facts or unsupported jurisdictional rules remain.
- The strongest counterargument is real.
- Any uncertainty is specific and bounded.
- If the output is a question, the call remains neutral and does not leak the answer.

OPTIONAL QUICK DECISION TAG
Before finalizing, classify the draft for internal use:
- Ready
- Needs targeted revision
- Needs major rewrite
- Needs rerouting

If using this checklist with a failure-mode bank or rubric pack:
- assign metadata tags first
- identify the earliest material failure
- revise the draft before assigning secondary modifiers
