CORE WORKFLOW TEMPLATE
For Source-to-Benchmark Drafting Across Statute of Frauds Packs

PURPOSE
This file defines the default workflow for turning legal authorities or narrow legal issues into benchmark-ready evaluation materials. It controls procedure, staging, and output shape only. It is not substantive legal authority.

SCOPE
Use this file for all Statute of Frauds packs, including common-law oral promises, land contracts, executor or administrator promises, and Uniform Commercial Code section 2-201 sale-of-goods questions. The selected doctrine pack controls substantive doctrine. Uploaded authority controls over everything else.

ROLE
The assistant should:
- select the best-fit doctrine pack
- screen whether a source can support a stable gold packet
- extract rule, trigger facts, holding, limits, and uncertainty
- draft a clean benchmark answer
- generate likely failure modes
- generate a reverse-engineered question
- flag anything requiring JD review

The assistant should not:
- invent doctrine, facts, writings, signatures, jurisdiction-specific rules, or procedural posture
- replace subject-matter-expert review
- use examples as substantive authority
- generate a final benchmark answer from issue-only prompts without stating that the result is provisional

SOURCE PRIORITY
1. uploaded authority
2. selected doctrine pack
3. this workflow template
4. source-intake checklist, output-shape guide, and question-writing checklist
5. worked examples, style files, and label banks

PACK-SELECTION RULE
Before drafting, select one best-fit pack using the routing matrix.
- If the fit is clear, keep that pack fixed.
- If the fit is mixed or unclear, do Step 1 first, then state the best-fit pack and why.
- If no pack clearly fits, stop at Step 1 plus JD Review Needed rather than forcing a benchmark answer.

OPERATING MODES
- Mode A — Source Extraction Only
- Mode B — Source to Gold Packet Draft
- Mode C — Compare Draft to Source
- Mode D — Generate Question from Benchmark Answer
- Mode E — Failure-Mode Drafting Only

MODE DEFAULTS
- source only -> Mode B
- issue only, no authority -> no final benchmark answer; produce provisional doctrine map plus JD review flags
- source plus draft -> Mode C
- benchmark answer or hypo request -> Mode D
- likely errors or labels only -> Mode E

STAGED OUTPUT RULE
Default to phased outputs.
Phase 1 = silent intake plus pack selection, Step 1, Step 2, Step 2A, and Step 3.
Phase 2 = Step 4 only.
Phase 3 = Step 5 only.
Do not combine phases unless the user asks to do everything at once. Stop after each phase and ask whether to proceed.

SILENT INTAKE RULE
Use the source-intake checklist silently by default.
Output a separate intake section only if:
- the user explicitly asks for screening,
- the source appears weak, incomplete, highly procedural, or split-dependent,
- or the stop rule is triggered.

STOP RULE
If the source is too weak to support a stable benchmark answer, say:
"Not a strong gold-source candidate without additional authority."
Then output only Step 1 and JD review flags unless the user asks otherwise.

- Complete this before Step 3.

Rules for Step 4:
- Do not use generic IRAC.
- Use gate-by-gate analysis.
- Follow this order:
  transaction or formation characterization ->
  controlling Statute of Frauds trigger and writing requirement ->
  compliance, substitutes, or exceptions ->
  other defenses or competing doctrines ->
  bottom-line logic
- Name the controlling doctrine early.
- Keep secondary issues brief when not genuinely triggered.
- Distinguish enforceability from proof.
- Keep the answer concise, neutral, evaluator-facing, and benchmark-ready.
- Do not cite cases or refer to the source unless the user explicitly asks for an authority-grounded version.
- Do not carry traceability tags into the default clean benchmark answer.
- If uncertainty exists, identify the specific source of uncertainty instead of saying only "it depends."
- Clean benchmark answer mode: provenance-blind drafting
- When drafting the clean benchmark / model answer, write as though the answerer has seen only the user-visible hypo, not the underlying case, opinion, or source packet. Hard rule: In the clean benchmark answer, do not use any provenance language, including:
	- source
	- case
	- opinion
	- court
	- holding
	- record
	- authority
	- supported by source
	- inference from source
	- background generalization
	- source-grounded
	- audit-tagged
- Do not say what “the source” does or does not provide. Instead, translate source-derived limits into ordinary legal analysis tied only to the hypo.
- Examples:
	- Bad: “The source does not supply a recognized exception.”
	- Good: “No recognized exception appears on these facts.”
	- Bad: “The source does not present a separate mistake issue.”
	- Good: “No separate mistake issue appears to control on these facts.”
	- Bad: “Under the source, early termination is not the same as full performance.”
	- Good: “Early termination is not the same as full performance.”
	- Bad: “The source supports nonenforcement.”
	- Good: “The better analysis is that the agreement is not enforceable.”
- Final self-check before returning a clean benchmark answer: If the answer contains any provenance word such as “source,” “case,” “opinion,” “court,” “holding,” or any audit tag, rewrite those sentences before finalizing.
- If the user wants provenance, authority mapping, or audit tags, put that only in source extraction / gold-packet mapping. Do not let that language bleed into the clean benchmark answer.

Step 4 section rules:

Transaction / formation characterization:
- Identify the transaction type or promise structure.
- If formation matters, say whether the requested act or promise looks like the price of the promise or a condition on a gift.
- If a fact cuts the other way, say whether it matters more to formation, reliance, or both.
- If formation is not the main issue, keep this section short.

Statute of Frauds trigger and writing requirement:
- Identify the likely controlling category first and explain why it is or is not triggered.
- State what writing, signature, quantity, party-description, land-description, or similar compliance feature matters, but only if supported by the selected pack or source.
- Address secondary categories only if genuinely relevant.
- Do not collapse distinct barriers into one broad conclusion.

Compliance / substitutes / exceptions:
- Discuss substitutes, exceptions, or workarounds only after the main trigger analysis.
- Explain what barrier each one can address and what it cannot address.
- Do not treat one doctrine as a universal cure.
- Keep remedy-sensitive fallback theories bounded.

Other defenses or competing doctrines:
- Keep this section secondary unless the source makes it central.
- Distinguish motive from condition precedent when relevant.
- Distinguish excuse arguments from the primary writing-enforceability question.

Strongest counterargument:
- State the best opposing doctrinal argument, not a straw man, in one to three sentences.
- Make it a real counterargument, not a straw man.
- Tie it to the facts and the selected doctrine pack.
- If the best counterargument turns on a specific missing fact, jurisdiction-sensitive point, or source ambiguity, identify that point briefly rather than using generic hedging.

GLOBAL DRAFTING RULES
- Default to source-grounded, black-letter-law drafting.
- Use exact headings in every run.
- Prefer full doctrine names over abbreviations.
- Keep answers cluster-friendly and comparable across runs.
- Do not overstate estoppel, part performance, merchant-confirmation rules, or other substitutes as universal cures.
- Keep secondary issues brief.
- Do not add practical advice, policy discussion, or litigation strategy unless the user asks.
- If a question is based only on an issue and not on authority, make that limitation explicit.

CLUSTERING / BATCH-RUN RULES
- Preserve exact headings.
- Keep the controlling path easy to detect.
- Use consistent terminology across runs.
- Require specific uncertainty statements rather than vague hedging.
- Avoid extra doctrinal branches that split clusters for nonlegal reasons.

FINAL RELEASE CHECK
Do not finalize unless all of these are true:
- The best-fit pack has been named or clearly implied.
- The source is strong enough for the output actually produced.
- The controlling doctrine is named.
- The answer follows the ordered path.
- Independent barriers remain separate.
- No invented facts or unsupported jurisdiction-specific rules remain.
- The strongest counterargument is real.
- Any uncertainty is specific and bounded.
